# Concrete Skills

Source: Dave Hoover and Adewale Oshineye, *Apprenticeship Patterns* (2009) — provenance: author-verified

## When it applies

- Competence needs demonstrating — job hunting, joining a team, earning a reviewer's trust
- "What can you do?" gets answered with things read about rather than built

## The rule

Identify 3–5 directly demonstrable skills relevant to the work you want, and for each build something you can show: a repo, a deployed app, a real bug fix. Answer "what can you do?" with proof, not aspiration — "I built X" beats "I'm familiar with X." Update the list as context changes; Year 1's concrete skill is Year 3's table stakes.

## Anti-patterns

- **Familiarity inflation** — "familiar with Kubernetes," never ran a cluster
- **Buzzword padding** — technologies from a job description, not from use
- **Aspiration as capability** — "I want to learn X" filed under "I can do X"
- **AI-borrowed proof** — pointing at code an AI wrote as evidence of your skill

## In the agentic era

The pattern's fifth rule became its center: your concrete skills are what you can do *without* the agent, and the honest ledger of that distinction is the skill inventory that matters now. When ability is claimed that couldn't be demonstrated cold, or agent output is about to be presented as personal proof, name **AI-borrowed proof** — the code may run; the skill wasn't built.
